# What is ROS 2?

ROS 2 (Robot Operating System 2) is not an operating system.

It is a robotics middleware that allows many independent programs (called nodes) to communicate with each other.

Instead of writing one huge program that controls an entire robot, ROS divides the robot into many smaller programs.

Example:

Camera Node
↓

Object Detection Node
↓

Navigation Node
↓

Motor Controller Node

Each node performs one task and communicates using ROS topics, services, or actions.

## src/

The **src** folder stores every ROS package that belongs to the project.

Nothing is built here.

Only source code exists here.

---

## build/

Created automatically after running:

colcon build

Purpose:

Stores temporary compilation files.

Think of it as the compiler's working directory.

---

## install/

Also created automatically.

Purpose:

Contains the installed version of every package.

When we execute:

source install/setup.bash

ROS loads packages from this directory.

Without sourcing this directory ROS cannot find our package.

---

## log/

Stores build logs.

Useful when debugging compilation problems.

# Why "robot_bringup"?

The word **bringup** is commonly used in robotics.

It means:

"Start everything needed to make the robot operational."

In large robots the bringup package launches:

* sensors
* controllers
* cameras
* localization
* robot state publisher
* navigation

In Phase 1 our bringup package only launches two simple nodes.

Creates a ROS node that publishes information.

Node name:

hello_robot

/robot/status

Message type:

std_msgs/String

Publishing frequency:

1 Hz

Published message:

Hello from VisionNav

---

## What happens internally?

Every second:

1. Timer triggers.
2. A String message is created.
3. Message data is filled.
4. Message is published.
5. Any subscriber immediately receives it.

---

# Subscriber Node

File:

status_listener.py

Purpose:

Listens to the topic:

/robot/status

Whenever a new message arrives:

The callback function executes automatically.

Example output:

Received: Hello from VisionNav

---

# Topic

Topic:

/robot/status

A topic is like a radio channel.

Anyone can broadcast.

Anyone can listen.

Publishers never know who is listening.

Subscribers never know who is publishing.

This loose coupling makes ROS scalable.

---

## Phase 3 – Gazebo Simulation

📌 Overview

Phase 3 of VisionNav moves the robot from static visualization (RViz) into physics simulation using Gazebo Classic. No SLAM or navigation is included in this phase.

The goal was to spawn the robot inside a simulated environment, get all onboard sensors publishing real data on ROS 2 topics, and get differential-drive motion actually working under physics.

🧠 What was implemented

✅ Robot Simulation Package

Created a dedicated ROS 2 package:

robot_simulation (ament_cmake)

Responsible for:

Gazebo world file
Robot spawn launch file
Controller configuration (diff_drive.yaml)

✅ Simulated World

File:

worlds/warehouse.world

An enclosed 10m x 10m room built from 4 static walls, plus 2 box obstacles inside it. Previously this was just an open ground plane with floating boxes — now it is a proper bounded environment.

✅ Gazebo Sensor Plugins

Added to visionnav_robot.urdf, bridging Gazebo's internal sensor data to ROS 2 topics:

lidar_link → /scan (sensor_msgs/LaserScan)
base_link (imu_sensor) → /imu/data (sensor_msgs/Imu)
rgb_camera_link → /camera/image_raw (sensor_msgs/Image)
depth_camera_link → /depth/points (sensor_msgs/PointCloud2) + /depth/image_raw

Each Gazebo sensor requires an explicit ROS plugin and topic remapping — without it, the sensor exists inside physics but is invisible to ROS 2.

✅ Differential Drive Control

Wired up gazebo_ros2_control so the existing ros2_control hardware interface in the URDF actually loads its controller configuration:

controller_manager loads diff_drive_controller
Config: config/diff_drive.yaml (wheel separation, wheel radius, odom frame)
Command topic: /diff_drive_controller/cmd_vel_unstamped
Odometry topic: /diff_drive_controller/odom

Verified by publishing a Twist command directly and confirming odometry position changed accordingly, then by driving the robot manually with teleop_twist_keyboard (remapped to the controller's cmd_vel topic).

✅ Caster Wheel Fix

The robot originally had a single rear caster only, causing the front to lift under forward acceleration. Fixed by:

Adding a second caster at the front (caster_front_link), mirroring the existing rear caster (caster_rear_link)
Correcting caster mount height — the caster sphere radius (0.05m) differs from the drive wheel radius (0.07m), so both caster joints were lowered by 0.02m so the spheres actually reach the ground

Verified using IMU orientation: before the height fix, the robot rested with a constant ~0.038 rad tilt (casters not touching ground); after the fix, resting orientation is level (quaternion ≈ identity) and stays level under hard forward acceleration.

🧩 Launch Flow

File:

launch/spawn_world.launch.py

Order of operations:

1. Start Gazebo with the warehouse world
2. Start robot_state_publisher with the robot_description (URDF text, with the controller config path resolved at launch time)
3. Wait, then spawn the robot entity into Gazebo via spawn_entity.py
4. Wait, then spawn the diff_drive_controller via controller_manager

🖥 Verification Performed

Commands used:

ros2 topic list
ros2 topic echo /scan --once
ros2 topic echo /imu/data --once
ros2 topic echo /camera/image_raw --no-arr --once
ros2 topic echo /depth/points --no-arr --once
ros2 topic hz /camera/image_raw
ros2 topic hz /depth/points

All 4 sensor topics confirmed publishing live, physically sane data (real LiDAR ranges, gravity in the IMU z-acceleration, correctly sized image/point-cloud buffers).

🧠 Key Learning Outcome

Gazebo and ROS 2 are separate systems — a ROS plugin is the only bridge between simulated sensor data and ROS topics
A sensor with no plugin, or a plugin pointing at the wrong library, fails silently inside Gazebo, not in ROS
ros2_control's Gazebo plugin needs an explicit path to its YAML config; xacro-style $(find pkg) substitution does not work here unless resolved before Gazebo receives the URDF
Physical stability in simulation depends on every ground-contact geometry (wheels and casters) actually reaching the ground — small radius/height mismatches show up as real physics problems, not just cosmetic ones
